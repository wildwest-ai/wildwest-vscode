# Assignment: release.sh Push Gate

**From:** CD(RSn).Cpt  
**To:** TM(RHk).Cpt  
**Branch:** `chore/release-push-gate`  
**Priority:** High — blocks repeat push violations  
**Model:** Claude Haiku 4.5 (Copilot)

---

## Problem

`release.sh` runs to completion including `git push`. The final message implies push is done.
TM reads script completion as authorization.

**"yes to release" ≠ "yes to push."** These are separate gates. The script does not enforce that.
Two push violations resulted (v0.12.0 and v0.13.0). Both accepted. Fix prevents recurrence.

---

## Assignment

Fix `scripts/release.sh` so it:

1. **Stops before any `git push` or `git tag push`**
2. **Prints the exact commands to run**, e.g.:
   ```
   Release build complete. Awaiting push authorization from S(R).
   
   When authorized, run:
     git push origin main
     git push origin v0.X.Y
   ```
3. **Exits with a clear message** — "Awaiting push authorization from S(R)."
4. **Does NOT run the push automatically** — not even with a `--push` flag or confirmation prompt

The push step must require the devPair lead (S(R) in TM's case) to type the commands manually.
No amount of script automation should bridge that gap.

## Scope

- File: `scripts/release.sh`
- Change: Remove auto-push step; add stop + print + exit
- Out of scope: Any other release.sh changes; version bump logic; DONE.md format
