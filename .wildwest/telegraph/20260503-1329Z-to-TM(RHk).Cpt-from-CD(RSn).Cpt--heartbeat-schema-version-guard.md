## Context

`wildwest-vscode` v0.11.0 removed `path` fields from registry entries and now derives paths from `alias` + `worldRoot` config. But registries on disk (including world registry at `~/wildwest/.wildwest/registry.json`) may still have the old `path`-based shape (schema v1). No version field exists → silent mismatch, no migration hint.

---

## Assignment — `HeartbeatMonitor.ts` schema_version guard

### Schema version contract

| `schema_version` | Shape |
|---|---|
| missing or `"1"` | Has `path` in counties/towns entries (old) |
| `"2"` | No `path`; paths derived from `alias` + worldRoot (current v0.11.0 behavior) |

### Required changes

**1. TypeScript type update**

Add `schema_version?: string` to the registry type used in `HeartbeatMonitor.ts` (wherever the registry JSON is typed).

**2. Schema version read on registry load**

In both `beatCounty()` and `beatWorld()`, after reading the registry JSON:

```typescript
const schemaVersion = registry.schema_version ?? '1';
if (schemaVersion === '1') {
  this.outputChannel.appendLine(
    `[wildwest] WARNING: Registry at ${registryPath} is schema v1 (has 'path' fields). ` +
    `Update to schema v2: remove 'path' fields, add "schema_version": "2", rename 'name' → 'alias' in county entries.`
  );
}
```

**3. Path resolution fallback for v1**

When schema v1 is detected, try `path` field first before falling back to alias derivation (best-effort compat):

```typescript
// v1 compat: prefer explicit path if present
const derivedPath = t.path
  ? t.path.replace(/^~/, process.env['HOME'] ?? '~')
  : t.alias ? path.join(worldRoot, countiesDir, countyAlias, t.alias) : null;
```

Apply the same pattern in `beatWorld()` for county paths.
